## Features

- **Post Photos** — Upload images stored as BLOB in Oracle Database, with optional comment text
- **Geolocation Capture** — GPS coordinates (lat/lon) recorded automatically when posting
- **Timeline Feed** — Responsive Cards region displaying all posts in reverse-chronological order
- **Like / Unlike** — Toggle reaction on any post; one like per user per post enforced at the database level
- **Delete Posts** — Users can delete their own posts (enforced by `CREATED_BY = :APP_USER`)
- **Interactive Map** — Oracle APEX Map region visualizing all post and like locations
- **Progressive Web App (PWA)** — Installable on mobile devices; runs standalone like a native app

## Database Design

### Entity Relationship

```
SM_POSTS (1) ──────< SM_REACTIONS (N)
  ID (PK)              ID (PK)
  POST_COMMENT         POST_ID (FK → SM_POSTS.ID)
  FILE_BLOB            REACTION
  FILE_MIME            LAT, LON
  FILE_NAME            CREATED, CREATED_BY
  LAT, LON             UPDATED, UPDATED_BY
  CREATED, CREATED_BY  UNIQUE (POST_ID, CREATED_BY)
  UPDATED, UPDATED_BY
```

### Tables

```sql
-- Main posts table: stores text + photo (BLOB) + GPS coordinates
CREATE TABLE SM_POSTS (
    ID           NUMBER GENERATED BY DEFAULT ON NULL AS IDENTITY,
    POST_COMMENT VARCHAR2(4000 CHAR),
    FILE_BLOB    BLOB,
    FILE_MIME    VARCHAR2(255 CHAR),
    FILE_NAME    VARCHAR2(255 CHAR),
    LAT          NUMBER,
    LON          NUMBER,
    CREATED      DATE NOT NULL,
    CREATED_BY   VARCHAR2(255 CHAR) NOT NULL,
    UPDATED      DATE NOT NULL,
    UPDATED_BY   VARCHAR2(255 CHAR) NOT NULL,
    CONSTRAINT SM_POSTS_ID_PK PRIMARY KEY (ID)
);

-- Reactions/likes table: unique per user per post
CREATE TABLE SM_REACTIONS (
    ID         NUMBER GENERATED BY DEFAULT ON NULL AS IDENTITY,
    POST_ID    NUMBER,
    REACTION   VARCHAR2(16 CHAR),
    LAT        NUMBER,
    LON        NUMBER,
    CREATED    DATE NOT NULL,
    CREATED_BY VARCHAR2(255 CHAR) NOT NULL,
    UPDATED    DATE NOT NULL,
    UPDATED_BY VARCHAR2(255 CHAR) NOT NULL,
    CONSTRAINT SM_REACTIONS_ID_PK PRIMARY KEY (ID),
    CONSTRAINT SM_REACTIONS_CON UNIQUE (POST_ID, CREATED_BY)
);
```

### Audit Trail Triggers

Both tables use `BEFORE INSERT OR UPDATE` triggers that automatically populate audit columns using Oracle APEX session context — ensuring complete user accountability with zero application-layer code:

```sql
CREATE OR REPLACE TRIGGER sm_posts_biu
    BEFORE INSERT OR UPDATE ON sm_posts
    FOR EACH ROW
BEGIN
    IF inserting THEN
        :new.created    := sysdate;
        :new.created_by := coalesce(sys_context('APEX$SESSION','APP_USER'), user);
    END IF;
    :new.updated    := sysdate;
    :new.updated_by := coalesce(sys_context('APEX$SESSION','APP_USER'), user);
END sm_posts_biu;
```

## Key Technical Highlights

### 1. Timeline Query with Inline Aggregation

The Cards region uses a single SQL query that aggregates reaction counts and determines per-user like status in one pass:

```sql
SELECT
  p.id,
  p.created_by                              AS user_name,
  p.post_comment                            AS comment_text,
  p.file_blob,
  p.file_mime,
  apex_util.get_since(p.created)            AS post_date,
  (SELECT COUNT(*) FROM sm_reactions smr
    WHERE smr.post_id = p.id)               AS reactions,
  (SELECT 'user-has-liked' FROM sm_reactions smr
    WHERE smr.post_id = p.id
      AND created_by = :APP_USER)           AS user_reaction_css
FROM sm_posts p
ORDER BY p.created DESC
```

The `user_reaction_css` column value is applied as a CSS class on the like button. When the current user has liked a post, the class `user-has-liked` turns the icon red — with no additional JavaScript required.

### 2. APEX Actions Framework for Interactive Features

Like and delete actions are registered using the APEX Actions API, decoupling UI triggers from business logic:

```javascript
apex.actions.add([{
  name: "like",
  action: (event, element, args) => {
    apex.items.P1_ACTION_ID.value = args.id;
    apex.event.trigger(document, 'action-like');
  }
}, {
  name: "delete",
  action: (event, element, args) => {
    apex.items.P1_ACTION_ID.value = args.id;
    apex.event.trigger(document, 'action-delete');
  }
}, {
  name: "open-map",
  action: () => {
    apex.event.trigger(document, 'action-open-map');
  }
}]);
```

Dynamic Actions handle the server-side execution via AJAX callbacks, triggering partial page refresh of the Cards region without a full page reload.

### 3. Map Region with UNION Query

The Map region displays post and like locations from two tables in a single geographic layer:

```sql
SELECT DISTINCT lat, lon, created_by,
  apex_util.get_since(created) AS since
FROM (
  SELECT lat, lon, created_by, created FROM sm_posts
  UNION
  SELECT lat, lon, created_by, created FROM sm_reactions
)
```

### 4. Progressive Web App (PWA)

Enabled via APEX application settings (`progressiveWebApp: installable = true`). APEX auto-generates the Web App Manifest and Service Worker — no external tooling, no build pipeline needed.

## Installation

### Prerequisites

- Oracle APEX workspace (free at [apex.oracle.com](https://apex.oracle.com))
- Oracle Database (included with the APEX free tier)

### Quick Install

1. In APEX: **App Builder → Import**
2. Upload `apex-social-media.zip`
3. Click through: **Next → Install Application → Install Supporting Objects**

The database tables (SM_POSTS, SM_REACTIONS) and triggers are created automatically via the Supporting Objects mechanism — no manual SQL execution required.

If you only need the database objects without the APEX application:

```sql

```

## What I Learned

Working through this Oracle LiveLabs workshop deepened my understanding of several APEX capabilities that are directly applicable to enterprise development:

- **BLOB handling in APEX** is seamless — the platform manages file upload, secure storage, and image rendering without custom backend code or external storage services.
- **Cards Region** is one of APEX's most versatile components. Combining SQL aggregate subqueries with CSS class columns enables highly dynamic UIs with minimal code — a pattern applicable to dashboards, feeds, and monitoring tools.
- **APEX Actions Framework** provides a clean architectural pattern for event-driven UIs. Decoupling action registration (JavaScript) from execution (Dynamic Actions) scales well in complex applications.
- **PWA support built into APEX** removes a significant barrier for enterprise mobile deployment. No App Store process, no native build toolchain — just a toggle in application settings.
- **Trigger-based audit trails** using `SYS_CONTEXT('APEX$SESSION','APP_USER')` ensure full accountability at the data layer, independent of the application layer — a best practice I will apply to all future table designs.
- **Oracle APEX Map Region** enables geographic data visualization from Oracle Database using only SQL — no external mapping libraries or API keys required.
